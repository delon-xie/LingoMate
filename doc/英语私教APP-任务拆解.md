### 第一部分：需求细化 (Requirements Refinement)

在动手写代码前，我们需要明确App具体“长什么样”和“能干什么”。

#### 1. 核心功能 (P0优先级 - 必须实现)
*   **双模交互**：支持按住说话（Voice）和键盘输入文字（Text），并能无缝切换。
*   **流式对话体验**：AI 老师的回复必须是“流式”的，即像打字机一样逐字显示文字，同时语音也要能够边生成边播放，降低等待焦虑。
*   **启发式单词教学**：遇到不会的单词，通过自然语言对话引导（解释 -> 举例 -> 提问造句），而不是直接抛出中文翻译。
*   **对话历史记录**：保留上下文记忆，用户可以回顾之前的对话。

#### 2. 辅助功能 (P1优先级 - 核心体验增强)
*   **随点随查**：在文字聊天记录中，点击任何一个英文单词，直接触发 AI 老师对该词的启发式讲解。
*   **生词本/记忆库**：自动提取 AI 重点讲解过的单词，存入本地，下次对话时 AI 会随机抽查。
*   **情景设定**：用户可以在首页选择场景（如：雅思口语模拟、日常闲聊、职场面试）。

### 第二部分：系统设计 (System Design)

采用 **C/S 架构 (Client-Server)**，前端在 iPhone 端，后端在你的本地电脑/局域网服务器上。

#### 1. 架构图解
*   **iOS 前端 (SwiftUI + MVVM)**
    *   **UI 层**：聊天气泡视图、录音按钮、单词高亮组件。
    *   **音频处理模块**：`AVAudioEngine`（支持回声消除 AEC，防止 AI 说话时被麦克风录入）、iOS原生 `SFSpeechRecognizer`（本地语音转文字，降低后端压力）。
    *   **网络层**：`URLSession` (WebSocket)，用于与本地电脑保持双向实时通讯。
    *   **本地存储**：`CoreData` 或 `SwiftData`，存储历史对话和生词本。
*   **本地后端 (Python FastAPI + WebSocket)**
    *   **网关层**：处理 WebSocket 连接，接收前端发来的文本。
    *   **AI 调度层**：管理 System Prompt，拼接对话上下文。
    *   **大模型层**：通过 API 调用本地部署的 **Ollama**（运行 Llama3 或 Qwen 模型）。
    *   **TTS 合成层**：使用本地语音合成模型（Edge-TTS），将文字转为音频流返回给手机。
    *  其他模型：如 ChatTTS（对GPU要求过高，总8G内存不够用） 、Piper（过于机械） 

#### 2. 核心交互时序图 (逻辑)

1. 客户端APP包括： iOS 端/macOS 桌面端
2. 用户在 客户端APP 说话 -> 手机端直接将语音转成纯文本 (STT)。
3. 手机通过 WebSocket 将文本发送给本地电脑后端。
4. 后端将文本送入大模型，大模型开始**流式**吐出回复词元 (Tokens)。
5. 后端积累到一整句话（遇到标点符号）时，将这句话送入 TTS 模块合成音频片段。
6. 后端将音频片段和文字同时下发给 客户端APP。
7. 客户端APP立刻播放音频并显示文字（实现极低延迟）。
8. 当出现，手机再次通过 WebSocket 将文本发送给本地电脑后端 ，后端回吐终止

#### 3 一些功能性需求

1. **强制锁定**为口语化音色，并自动处理中英文切换。
2. 用户可以通过语音或文字说：大一点声、小一点声、快一点说、慢一点说，进行播放音量和语速控制
3. 客户端可以配置：语速、声音大小、音色
4. voice = "zh-CN-XiaoyiNeural"  # 中文首选聊天音色
5. voice = "en-US-AriaNeural"    # 英文首选对话音色
6. 中英文可以自动切换
7. 后端检查 qwen2.5:3b 模型是否在运行，需要保证它的正确状态

### 第三部分：任务拆分 (WBS - Work Breakdown Structure)

我们将任务分为 **前端 (iOS/ macOs desktop)**、**后端 (Backend)** 和 **模型 (AI)** 三条线。

#### 🔹 1. 前端开发任务 (iOS/ macOs desktop)
*   **UI-01**: 搭建主界面（情景选择页、聊天对话页、生词本页）。
*   **UI-02**: 实现高仿微信/ChatGPT的对话气泡组件，支持打字机效果。
*   **Audio-01**: 集成麦克风权限与录音功能，必须开启回声消除。
*   **Audio-02**: 集成 `SFSpeechRecognizer`，实现说话实时转文字。
*   **Audio-03**: 实现音频流播放器，能够拼接并播放后端传来的多个音频片段。
*   **Net-01**: 封装 WebSocket 客户端，处理连接、重连、发送、接收流式数据。
*   **Data-01**: 设计并实现 `SwiftData` 数据库模型（Message, Vocabulary）。

#### 🔹 2. 后端开发任务 (Python)
*   **API-01**: 搭建 FastAPI 基础框架，配置局域网跨域 (CORS) 与 IP 访问。
*   **API-02**: 实现 WebSocket 路由，维护客户端连接状态。
*   **LLM-01**: 封装大模型调用接口（通过 langchain 或直接调用 Ollama API）。
*   **TTS-01**: 调研并集成一个本地 TTS 模型，实现“输入文本 -> 输出音频流”。
*   **Core-01**: 编写中控逻辑，实现“边收字、边断句、边合成语音、边发送”的并发管线 (Pipeline)。

#### 🔹 3. 模型与提示词工程 (AI)
*   **AI-01**: 在本地电脑安装部署 Ollama 及选定的大语言模型。
*   **AI-02**: 编写并调试核心 **System Prompt**（包括输入水平自适应理论 $$i+1$$、纠错逻辑、启发式教学逻辑）。
*   **AI-03**: 编写“单词专项讲解”的特定 Prompt（当用户点击单词时触发）。

### 第四部分：开发计划任务 (Sprint Plan)

假设由你一人全栈开发，以 **周 (Week)** 为单位，推荐采用敏捷迭代的方式，先跑通最简核心，再加“精装修”。

#### 🚩 Sprint 1：基础设施与跑通文字闭环 (第1周)
*   **目标**：手机能用文字和本地电脑里的大模型聊天。
*   **任务**：
    *   [模型] 本地安装 Ollama，下载并测试大模型运行状态。
    *   [后端] 用 FastAPI 写出 WebSocket 接口，能把收到的消息发给大模型，并把大模型的文字流式返回。
    *   [前端] 用 SwiftUI 画出基本的聊天界面。
    *   [前端] 实现 WebSocket 连通本地局域网 IP，实现文字收发和打字机 UI。

#### 🚩 Sprint 2：语音注入，实现“听说” (第2周)
*   **目标**：嘴巴能说，耳朵能听。
*   **任务**：
    *   [前端] 接入 iOS 本地语音转文字 (STT)，实现按住说话，松开发送纯文本给后端。
    *   [后端] 部署本地 TTS 引擎。
    *   [后端] 编写并发逻辑：收到 LLM 一句话 -> 转交 TTS -> 返回音频文件给前端。
    *   [前端] 接收后端的音频数据并播放。

#### 🚩 Sprint 3：注入“老师的灵魂”与体验优化 (第3周)
*   **目标**：让它从一个机器人变成“启发式英语外教”。
*   **任务**：
    *   [模型] 反复打磨 System Prompt。测试用户说中文或故意犯错时，AI 能否遵循“不直接翻译、启发造句”的规则。
    *   [后端] 优化流式语音的延迟（关键技术点：句号截断合成音频）。
    *   [前端] 解决 AI 说话时，麦克风的回声干扰问题 (AEC)。

#### 🚩 Sprint 4：核心功能深化 (第4周)
*   **目标**：实现单词点按讲解与历史记录。
*   **任务**：
    *   [前端] 改造文字气泡 UI，使每个英文单词可以独立点击。
    *   [前端] 点击单词后，发送特殊指令给后端（如 `[CMD:TEACH_WORD:apple]`）。
    *   [后端] 识别特殊指令，调用特定的“教学 Prompt”进行回复。
    *   [前端] 引入 SwiftData，将聊天记录持久化存入手机，杀掉后台重新打开依然存在。

#### 🚩 Sprint 5：生词本与最终打磨 (第5周)
*   **目标**：完善教育闭环。
*   **任务**：
    *   [后端/前端] 每次对话结束后，后端自动总结出用户不会的单词，发给前端。
    *   [前端] 制作“生词本”页面列表。
    *   [前端] UI 细节打磨（动画效果、加载状态、错误提示）。
    *   **最终局域网环境真机验收！**
